## 1. エグゼクティブサマリー

本SRSは、MUSUBIの次期メジャーバージョン（v3.0.0）に向けた機能拡張要件を定義します。AIコーディングツールの競合調査（OpenHands、Kiro、Spec Kit、Cline）に基づき、MUSUBIの競争力強化のために22の要件を優先度別に整理しました。

### 1.1 スコープ

- コミュニティ成長施策（認知度向上、マーケティング）
- **OpenHands由来のコア機能**（スタック検出、メモリ圧縮、セキュリティ分析）
- 機能差別化（ブラウザ自動化、Web GUI、VS Code拡張）
- エコシステム拡大（多言語対応、CI/CD統合）
- 長期的改善（ローカルLLM、コスト追跡）

### 1.2 優先度定義

| 優先度 | 目標 | 説明 |
|--------|------|------|
| P0 | v2.2.0 (2026年Q1) | クリティカル：コミュニティ成長施策 |
| P0-B | v2.2.0 (2026年Q1) | クリティカル：OpenHands由来コア機能 |
| P1 | v2.5.0 (2026年Q2) | 高：機能差別化 |
| P2 | v3.0.0 (2026年Q3) | 中：エコシステム拡大 |
| P3 | v3.5.0 (2026年Q4) | 低：長期的改善 |

## 2. P0要件（クリティカル：コミュニティ成長）

MUSUBIは技術的に成熟していますが、競合と比較してGitHubスター数が少ない状況です（現在2スター、Cline: 52K、OpenHands: 65K）。認知度向上がv3.0.0に向けた最重要課題です。

### REQ-P0-001: GitHubリポジトリ最適化

**ID**: REQ-P0-001
**優先度**: P0（クリティカル）
**パターン**: Ubiquitous

**要件**:
システムは、以下の最適化された発見可能性を持つ包括的なGitHubリポジトリを提供しなければならない：

1. 適切なトピックタグ（ai-coding、sdd、specification-driven-development、claude-code等）
2. SEO最適化されたリポジトリ説明（英語・日本語）
3. ビジュアルREADMEバッジ（CI/CD、npm、License、Downloads）
4. デモGIF/スクリーンショット付きREADME

**受入基準**:
- [ ] GitHub検索で「sdd tool」「ai coding agent」でトップ10に表示
- [ ] READMEにアニメーションデモGIFを含む
- [ ] 指定の4つのバッジが表示されている
- [ ] 英語・日本語両方の説明が存在

**トレーサビリティ**: TST-P0-001

## 3. P0-B要件（クリティカル：OpenHands由来のコア機能）

OpenHandsの分析に基づき、MUSUBIの基盤品質を向上させるコア機能を定義します。これらはv2.2.0で実装する優先度の高い機能です。

### REQ-P0-B001: スタック検出システム（Stuck Detector）

**ID**: REQ-P0-B001
**優先度**: P0-B（クリティカル）
**パターン**: State-Driven
**出典**: OpenHands `openhands/controller/stuck.py`

**要件**:
AIエージェントがタスクを実行している間、システムはスタック状態（無限ループ、繰り返しエラー）を自動的に検出し、介入しなければならない。

**機能要件**:
1. **検出シナリオ**:
   - 同じアクション・同じ結果の繰り返し（4回以上）
   - 同じアクション・エラーの繰り返し（3回以上）
   - モノローグ状態（出力なしの思考継続）
   - コンテキストウィンドウ超過ループ
   - ステージ間の無限往復

2. **介入アクション**:
   - ユーザーへの警告通知
   - 代替アプローチの自動提案
   - セッション状態の保存と中断

3. **統合ポイント**:
   - `musubi-workflow` CLI との統合
   - 各ステージ（要件/設計/実装/テスト）での検出

**受入基準**:
- [ ] 4回連続の同一アクション繰り返しを検出できる
- [ ] エラーループ検出後に代替アプローチを提案する
- [ ] スタック検出時にユーザーに通知される
- [ ] 検出履歴が `steering/memories/` に記録される

**トレーサビリティ**: TST-P0-B001

### REQ-P0-B002: キーワードトリガー型スキル

**ID**: REQ-P0-B002
**優先度**: P0-B（クリティカル）
**パターン**: Event-Driven
**出典**: OpenHands `skills/README.md`、`openhands/microagent/`

**要件**:
会話中に特定のキーワードが出現した場合、システムは関連するスキルを自動的に活性化しなければならない。

**機能要件**:
1. **トリガー定義**:
   - 各スキルに `triggers` フィールドを追加
   - 複数キーワードのサポート（OR条件）
   - 正規表現パターンのサポート

2. **スキル活性化**:
   - キーワード検出時に関連スキルのプロンプトを注入
   - 複数スキルの同時活性化（優先度順）
   - 活性化ログの記録

3. **設定形式**:
   ```yaml
   name: testing-skill
   triggers:
     - test
     - unit test
     - テスト
     - 単体テスト
   ```

**受入基準**:
- [ ] スキル定義ファイルに `triggers` フィールドが追加される
- [ ] 「テストを書いて」で testing スキルが自動活性化する
- [ ] 活性化されたスキルがログに記録される
- [ ] 日本語/英語両方のキーワードをサポート

**トレーサビリティ**: TST-P0-B002

### REQ-P0-B003: リポジトリ固有スキル

**ID**: REQ-P0-B003
**優先度**: P0-B（クリティカル）
**パターン**: Event-Driven
**出典**: OpenHands `.openhands/skills/` ディレクトリ

**要件**:
プロジェクトが `.musubi/skills/` ディレクトリを持つ場合、システムはリポジトリ固有のスキルを自動的にロードしなければならない。

**機能要件**:
1. **ディレクトリ構造**:
   ```
   .musubi/
   ├── skills/
   │   ├── repo.md           # リポジトリ概要（自動読み込み）
   │   ├── testing.md        # テスト規約
   │   └── deployment.md     # デプロイ手順
   └── config.yml            # 設定ファイル
   ```

2. **読み込み順序**:
   - グローバルスキル（MUSUBI内蔵）→ リポジトリスキル
   - 同名スキルはリポジトリ版が優先

3. **自動生成**:
   - `musubi-onboard` 実行時に `repo.md` を自動生成
   - 既存の `.github/`, `.vscode/` 設定を分析

**受入基準**:
- [ ] `.musubi/skills/` ディレクトリのスキルが自動読み込みされる
- [ ] `repo.md` がセッション開始時にコンテキストに含まれる
- [ ] `musubi-onboard` が `repo.md` を自動生成する
- [ ] リポジトリスキルが内蔵スキルを上書きできる

**トレーサビリティ**: TST-P0-B003

### REQ-P0-B004: メモリコンデンサー

**ID**: REQ-P0-B004
**優先度**: P0-B（クリティカル）
**パターン**: State-Driven
**出典**: OpenHands `openhands/memory/condenser/`

**要件**:
コンテキストウィンドウが設定閾値を超えている間、システムは古いイベントを自動的に要約・圧縮しなければならない。

**機能要件**:
1. **圧縮戦略**:
   - 古いイベントから順に要約
   - ユーザーメッセージ間のエージェントアクションを要約
   - 重要なアーキテクチャ決定は保持

2. **設定オプション**:
   ```yaml
   condenser:
     type: llm              # llm | recent | noop
     max_size: 100          # 履歴の最大イベント数
     keep_first: 2          # 常に保持する最初のイベント数
     summary_model: haiku   # 要約用LLM
   ```

3. **メタデータ保存**:
   - 圧縮履歴を `steering/memories/` に保存
   - 圧縮前後の差分情報

**受入基準**:
- [ ] コンテキスト超過時に自動的に履歴が圧縮される
- [ ] 圧縮後もセッションの継続性が維持される
- [ ] 重要な決定事項が圧縮で失われない
- [ ] 圧縮設定が `project.yml` で変更可能

**トレーサビリティ**: TST-P0-B004

### REQ-P0-B005: クリティック（評価）システム

**ID**: REQ-P0-B005
**優先度**: P0-B（クリティカル）
**パターン**: Event-Driven
**出典**: OpenHands `openhands/critic/`

**要件**:
各SDDステージが完了した場合、システムは出力品質をスコアリング（0.0-1.0）し、閾値未満の場合は再実行を促さなければならない。

**機能要件**:
1. **評価対象**:
   - 要件の完全性・明確性スコア
   - 設計のトレーサビリティスコア
   - 実装の憲法準拠スコア
   - テストのカバレッジスコア

2. **スコアリング基準**:
   ```
   0.0 - 0.3: 不合格（再実行必須）
   0.3 - 0.5: 要改善（警告付きで続行可）
   0.5 - 0.8: 合格
   0.8 - 1.0: 優秀
   ```

3. **出力形式**:
   - CLI出力にスコア表示
   - `steering/memories/quality_report.md` に履歴保存

**受入基準**:
- [ ] 各ステージ完了時にスコアが表示される
- [ ] スコア0.3未満で再実行が促される
- [ ] スコア履歴がレポートファイルに保存される
- [ ] `musubi-validate score` でプロジェクト全体スコアが表示される

**トレーサビリティ**: TST-P0-B005

### REQ-P0-B006: GitHub Issue自動解決

**ID**: REQ-P0-B006

**パターン**: Event-Driven
**出典**: OpenHands `openhands/resolver/`

   - 実装変更の生成
   - テスト追加
   - PR作成（Draft）

3. **トレーサビリティ**:
   - PR説明にIssue番号を自動リンク
   - 変更ファイルごとに要件IDを記載

**受入基準**:
- [ ] `musubi-resolve --issue 123` でIssueからPRが生成される
- [ ] GitHub Actionsワークフローが提供される
- [ ] 生成されたPRにトレーサビリティ情報が含まれる
- [ ] Draft PRとして作成される（自動マージなし）

**トレーサビリティ**: TST-P0-B006

### REQ-P0-B007: セキュリティリスクアナライザー

**ID**: REQ-P0-B007
**優先度**: P0-B（クリティカル）
**パターン**: State-Driven
**出典**: OpenHands `openhands/security/`

**要件**:
コード生成やコマンド実行が行われている間、システムはセキュリティリスクを評価し、高リスクアクションは確認を求めなければならない。

**機能要件**:
1. **リスクレベル**:
   - LOW: 自動承認
   - MEDIUM: ログ記録、続行
   - HIGH: ユーザー確認必須

2. **検出パターン**:
   - ハードコードされたシークレット
   - 危険なシェルコマンド（rm -rf、sudo等）
   - SQLインジェクション脆弱性
   - 未検証のユーザー入力使用

3. **確認モード**:
   ```yaml
   security:
     confirmation_mode: true
     risk_threshold: MEDIUM
   ```

**受入基準**:
- [ ] ハードコードされたシークレットが検出・警告される
- [ ] 危険コマンドの実行前に確認が表示される
- [ ] リスク評価結果がログに記録される
- [ ] 確認モードの有効/無効が設定可能

**トレーサビリティ**: TST-P0-B007

### 9.1 EARS形式リファレンス

| パターン | 構文 | 使用場面 |
|----------|------|----------|
| Ubiquitous | システムは〜しなければならない | 常に適用される要件 |
| Event-Driven | 〜の場合、〜しなければならない | 特定イベントで発動する要件 |
| State-Driven | 〜している間、〜しなければならない | 特定状態で適用される要件 |
| Optional Feature | 〜を好む場合、〜しなければならない | オプション機能の要件 |
| Unwanted Behavior | 〜の場合、〜してはならない | 禁止される動作 |
